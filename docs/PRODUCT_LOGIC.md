# Product Logic

## 1. 单一用户心智

用户只需要理解三件事：

- 自动备份开不开
- 系统什么时候醒来检查
- 距离上次成功备份至少多久，才允许真正执行

不要让用户理解内部同步器、双写配置或“先改这个再发布那个”的工程概念。

## 2. 为什么用“每周检查 + 两周最短间隔”

“两周一次”的产品含义不是“直接写一个每 14 天的定时任务”。

采用的逻辑是：

- 定时任务负责在固定时间醒来
- 备份策略负责判断距离上次成功备份是否至少 14 天

这样做有几个优点：

- 定时任务配置简单稳定
- 用户容易理解
- 某次失败时，不需要人工重排日历；下一个检查点会自动补跑

## 3. 本机与 NAS 为什么分两种存储

本机：

- 目标是便于频繁恢复
- 每份快照都是完整视图
- 存储层使用硬链接复用未变化文件

NAS：

- 目标是离机归档
- 每次保留完整归档版本
- 实际支持任何稳定可写的 SMB 挂载目录，不要求目标设备一定是 NAS
- 采用先打包再上传，避免直接用 SMB/NFS 逐小文件复制带来的慢和不稳定

## 4. 什么变化会触发备份

系统默认会忽略高频运行态噪音，例如：

- 日志
- agent 会话流水
- 微信同步状态文件
- 备份系统自己的状态文件

但正常业务产物会参与变化判断，例如：

- 每日记忆
- 每日总结
- 其他正式产出的工作文件

## 5. 为什么提醒只看备份仓库自身大小

容量提醒应该只回答用户真正关心的问题：

- 备份仓库自己占了多大
- 是否该删一部分早期版本

它不应该：

- 扫描整台 NAS
- 试图代表用户分析无关目录
- 读取和备份功能无关的其他设备内容

## 6. 为什么正常检查不打扰用户

自动任务的正确体验是：

- `BACKUP_OK`
- `BACKUP_SKIPPED`

都静默处理。

只有这些情况才回传消息：

- `BACKUP_FAILED`
- `BACKUP_ALERT`

而且 `BACKUP_ALERT` 也只在真正需要提醒时才发送，不重复轰炸。
